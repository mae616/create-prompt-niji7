# create-prompt

niji journey用プロンプトを生成するスキル。

## トリガー
- ユーザーが `/create-prompt` を実行したとき
- ユーザーが「プロンプトを作って」「prompt生成」等を依頼したとき

## 説明
入力情報からniji journey向けの画像生成プロンプトを、ドメインカプセルの知識を合成して生成する。
GPT/Geminiのカスタム指示と違い、ドメインごとにカプセル化された指示を矛盾なく合成するのが特徴。

## 入力
- `prompts/input.md` の内容（ユーザーが記述した入力情報）
- または、スキル実行時の引数テキスト

## 実行手順

### 1. 入力の読み取り
- `prompts/input.md` を読む
- 引数がある場合はそちらを優先

### 1.5. キャラクター設定の参照
- 入力テキストに含まれる名前が `characters/` 配下のMDファイル名と一致するか確認する
  - 例: 「しずくaiが団子を食べている」→ `characters/しずくai.md` を探す
- ファイルが見つかった場合、そのキャラクターの外見設定を入力に統合する
  - キャラクター設定の外見（髪色・目色・衣装等）をプロンプトの素材情報として使用
  - 入力で明示的に上書きされた項目はそちらを優先する
  - キャラクター設定に含まれない情報（構図・背景・雰囲気等）は入力テキストに従う
- ファイルが見つからない場合はスキップ（キャラクター未登録として通常フローへ）
- `_template.md` は除外

### 2. ドメインカプセルの読み込み
- `.claude/skills/create-prompt/domains/` 配下の全 `.md` ファイルを読む
- `_template.md` は除外（テンプレートファイル）
- 各カプセルの `priority`（優先度）と `conflicts`（競合ルール）を確認

### 3. プロンプト合成

#### 現在のカプセル一覧（priority順）
| priority | カプセル | 役割 | 元の段階 |
|----------|---------|------|---------|
| 10 | output-rules.md | 合成の優先順位・全体方針・プロンプト形式 | 段階3 |
| 9 | face-expression.md | 顔・表情・目元構造制御 | 段階3 |
| 8 | art-style.md | 画風・線質・色面構造・線構造定義 | 段階3 |
| 8 | character-form.md | キャラクター造形・絵柄ポジション | 段階3 |
| 7 | coloring.md | 色彩・配色・影色・色相バランス制御 | 段階3 |
| 7 | painterly-texture.md | 絵画的質感・画材・技法表現 | 段階3 |
| 5 | composition.md | 情報密度・視線誘導・空間設計 | 段階3 |
| 4 | character.md | キャラクター外観基礎（素材情報） | 段階2 |
| 3 | world-setting.md | 世界観・環境基礎 | 段階1 |

#### プロンプト構築順序（output-rules.mdの優先順に準拠）
```
0. [描画表現/画風] → 1. [世界観] → 2. [キャラ設定] → 2.5. [空間設計] → 2.6. [コンセプト/視覚ストーリー] → 3. [統合プロンプト]
```

#### 統合プロンプトのフレーズ順（固定）
```
1.主役物理定義 → 2.焦点物体 → 3.髪 → 4.目 → 5.肌 → 6.衣装層 → 7.ポーズ+カメラ → 8.光 → 9.画材 → 10.技法 → 11.色面構造 → 12.影色混合 → 13.背景 → 14.密度差 → 15.線の役割 → 16.禁止要素
```

#### 合成ルール
- output-rules.mdの優先順位（画風成立 > 主役視認性 > 配色成立 > 構造維持 > 情報再現）を常に守る
- `conflicts` で定義された競合がある場合、`priority` が高いカプセルを優先
- カプセルが存在しないドメインはスキップ
- 入力情報で明示的に指定されたものはカプセルのデフォルトを上書き
- 曖昧な雰囲気語・ジャンルラベルは使わず、視覚的に解釈可能な要素へ分解する

### 4. 履歴ファイル作成（初回生成時）
`prompts/history/YYYY-MM-DD_HHmm_{title}.md` を新規作成する。
- `{title}` は入力情報から短いタイトルを自動生成（英数字・ハイフン）
- テンプレートは `prompts/history/_template.md` に準拠
- ファイルパスをユーザーに通知する

#### 履歴ファイルの構造
```markdown
# {タイトル}

> 作成: YYYY-MM-DD HH:mm
> 最終更新: YYYY-MM-DD HH:mm
> ステータス: 作業中 / 完成

---

## 入力

### 描きたいもの
{入力情報をここに記録}

### スタイル指定（任意）
### 参考（任意）

---

## v1

### 判断メモ
- 使用カプセル: {一覧}
- 画風判断: {選択理由}
- 競合解決: {あれば}

### English Prompt
（英語プロンプト）

### 日本語プロンプト
（日本語プロンプト）

---

## v1 → v2（修正指示）
> {ユーザーの修正指示}

---

## v2
### 変更点
- {差分}

### English Prompt / 日本語プロンプト
（以降繰り返し）
```

### 5. 出力と記録
- 生成したプロンプトは画面に表示すると同時に、履歴ファイルに追記する
- 初回は「## v1」セクションとして記録
- 修正のたびにバージョンを上げて追記（v2, v3...）
- 修正指示もそのまま記録する（「## vN → vN+1（修正指示）」セクション）
- 履歴ファイルは上書きしない（追記のみ）

### 6. 継続対話
生成後、以下を提示：
```
プロンプトを生成したにゃ！
履歴: prompts/history/{ファイル名}.md

次にできること：
1. 「修正して」「〜を強調して」→ 修正版を生成（vN+1として記録）
2. 「バリエーション」→ 同じ入力で異なるアプローチを生成
3. 「完成」→ ステータスを「完成」に変更
4. 「新規」→ 新しい履歴ファイルで開始
```

### 7. 再開時の動作
ユーザーが既存の履歴ファイルを渡して `/create-prompt` を実行した場合：
- そのファイルの最新バージョンを読み取る
- 入力情報と直前のプロンプトをコンテキストとして継続
- 修正指示に基づいて次のバージョンを生成・追記する

## ドメインカプセルの仕組み

各カプセルは独立したMDファイルで、以下の構造を持つ：

```markdown
# {ドメイン名}

## メタデータ
- priority: {1-10、高いほど優先}
- conflicts: [{競合する可能性のあるドメイン名}]

## ルール
{このドメインのプロンプト生成ルール}

## キーワード
{このドメインで使用する代表的なキーワード/フレーズ}

## 例
{このドメインの出力例}
```

**カプセル化の利点:**
- 各ドメインの知識が独立 → 1つ変えても他に影響しない
- 競合解決ルールが明示的 → GPT/Geminiのカスタム指示より予測可能
- ドメイン単位で追加/削除が容易

## 出力フォーマット
- niji journeyの `/imagine prompt:` 形式に準拠
- 英語プロンプトが主（niji journeyの処理精度が高い）
- 日本語プロンプトは参照/確認用

## 品質チェック（生成後セルフレビュー）

プロンプト生成後、出力する前に必ず以下のチェックを通す。
不合格の項目がある場合は自動で修正してから出力する。

### A. カプセル準拠チェック

| # | チェック項目 | 参照カプセル |
|---|------------|------------|
| A1 | 概念語・ジャンルラベルが単体で使われていないか（必ず `defined as` / `visually interpreted as` 付き） | output-rules |
| A2 | 禁止ブロック4文が末尾に含まれているか（no photorealism / no heavy outlines / no pure black lines / no uniform digital shading） | output-rules |
| A3 | 画材が `[medium], visually interpreted as [physical behavior]` 形式か | painterly-texture |
| A4 | 技法が `[technique], defined as [visual construction behavior]` 形式か | painterly-texture |
| A5 | 色面構造の必須文が含まれているか（color-plane based composition 等） | art-style |
| A6 | 色相バランス必須文が含まれているか（color harmony controlled by... 等） | coloring |

### B. 画面成立チェック（優先順位順）

| # | チェック項目 | 優先度 |
|---|------------|--------|
| B1 | **画風成立**: 塗り方・線の扱い・描画密度が明示されているか | 最高 |
| B2 | **主役視認性**: 主役が1つに寄っているか。主役周辺の密度が整理されているか | 高 |
| B3 | **配色成立**: 無彩色のみ/白背景になっていないか。色面で空間が成立するか | 高 |
| B4 | **構造維持**: 空間タイプと光条件が指定されているか | 中 |
| B5 | **情報再現**: 入力の形状・構造的骨格が維持されているか | 低 |

### C. 禁止事項チェック

| # | チェック項目 |
|---|------------|
| C1 | photo / realistic / photorealistic / hyper-realistic / 3D render が含まれていないか |
| C2 | pure black / bold outline / heavy outline が使われていないか |
| C3 | digital art が単体で使われていないか（`visually interpreted as` なしで） |
| C4 | ジャンルラベル（moe, kemonomimi, furry, chibi, lolita, gothic 等）が単体で使われていないか |
| C5 | 写真由来の質感語（smooth skin, glossy, polished 等）が意図せず含まれていないか |

### D. フレーズ構造チェック

| # | チェック項目 |
|---|------------|
| D1 | 1フレーズ＝1視覚意図になっているか（複数の意図が混在していないか） |
| D2 | フレーズ順がoutput-rulesの優先順（1.主役物理定義〜16.禁止要素）に沿っているか |
| D3 | 意味衝突するフレーズが隣接していないか |
| D4 | 過剰な形容詞の重ねがけがないか |

## ガードレール（自律的な安全制御）

### 入力バリデーション
- 入力が空または曖昧すぎる場合 → 生成せず、最低限の情報（何を描きたいか）を質問する
- 入力に写実/フォトリアル系の要望がある場合 → カプセルの制約を説明し、イラスト表現内での代替案を提案する

### カプセル整合性
- カプセル間で矛盾する指示を検出した場合 → priority が高い方を採用し、判断メモに記録する
- カプセルが0個の場合（domains/ が空） → デフォルト動作せず、カプセル未設定であることを通知する

### 出力の安全性
- 生成したプロンプトが niji journey で規制される可能性のある表現を含む場合 → 該当箇所を指摘し、安全な代替表現を提案する
- 過度に長いプロンプト（概ね400語超）→ 冗長な箇所を指摘し、圧縮版も併記する

### 履歴ファイルの保護
- 既存の履歴ファイルを上書きしない（追記のみ）
- ステータスが「完成」のファイルに追記する場合 → 確認を取る
- 履歴ファイルの過去バージョンを削除・変更しない

### メモリ・慣性対策
- 新規生成時は直前のセッションの色相・画材・光条件を引き継がない
- 支配色相系統、素材カテゴリ、光の種類、画材カテゴリを毎回変更する
- 類似が検出された場合は色相を構造的にシフトする
